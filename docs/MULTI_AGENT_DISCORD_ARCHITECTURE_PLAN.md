# 技术总监 — 双层 AI 编排架构方案（Tmux V2.1）

## 概述

单一智能体“技术总监”作为**编排层**，持有业务上下文、设计决策、项目约束与失败经验；真正的落地工作由多个**执行层 Worker**完成。与第一版方案不同，V2.1 不再把 OpenClaw 内生的 `sessions_spawn` 作为主执行通道，而是让技术总监通过 `exec` 启动外部 `tmux` 会话，直接拉起独立的 CLI Worker。

```text
用户 / Boss
  ↓
技术总监（编排层）
  - 理解需求
  - 澄清范围
  - 写设计
  - 拆任务
  - 选模型
  - 跟踪状态
  - 失败重试
  - 汇报进展
  - 沉淀知识
  ↓
tmux + worktree + worker CLI（执行层）
  - 独立终端
  - 独立工作目录
  - 独立生命周期
  - 可旁路观察 / 打断 / 注入指令
```

这套结构的核心思想是：

- 技术总监负责“想清楚、安排清楚、验收清楚”；
- Worker 负责“把具体事情做完”；
- `tmux` 负责“让每个 Worker 像一个可观察、可打断、可恢复的终端工人”。

这不是单纯的“多开几个写代码的 Agent”，而是把 AI 从单一编码助手升级为**软件交付编排中枢**。

---

## 设计目标

这套架构的目标不只是提升编码效率，而是覆盖完整的软件开发与交付过程，包括但不限于：

- 需求澄清
- 技术方案设计
- 开发任务拆解
- 编码实现
- 测试与回归
- 代码评审
- PR 与 CI 跟踪
- 项目进展汇报
- 培训文档与使用手册编写
- 发布说明与复盘总结

换句话说，技术总监管理的对象不是“代码文件”本身，而是**整个研发过程中的任务、信息流和交付物**。

---

## 为什么从 `sessions_spawn` 升级到 Tmux

第一版设计基于 `sessions_spawn`，优点是干净、官方、配置统一；但在复杂编码任务和长链路交付任务里，存在几个明显短板：

- **黑盒执行**：子 Agent 跑起来以后，不如真实终端那样容易盯进度。
- **中途纠偏困难**：方向错了时，无法像操作终端一样即时“喊停 + 改指令”。
- **任务追踪弱**：Gateway 重启后，会话虽然可能还在，但编排层视角下的任务状态容易悬挂。
- **资源隔离一般**：长时构建、测试、安装等任务仍共享较多运行上下文。
- **交付类型单一**：更适合单轮执行，不够适合“实现 + 文档 + 汇报 + review”这种多产物链路。

Tmux V2.1 的思路是把 Worker 变成“外部可控进程”而不是“内部黑盒会话”：

- 编排层通过 `exec` 启动 Bash 脚本；
- Bash 脚本创建 `git worktree` + `tmux session`；
- `tmux` 里跑独立 CLI Worker；
- 技术总监可随时 `tmux send-keys`、`capture-pane`、`kill-session`；
- 不同类型的 Worker 可并行生成不同交付物，而不互相污染上下文。

这使整个系统更像一个真正的自动化研发流水线，而不是单纯的聊天机器人调机器人。

---

## Tmux 是什么，为什么这里必须用它

`tmux` 是一个**终端复用器**（terminal multiplexer）。你可以把它理解为“一个可后台持续运行、可随时接管、可随时旁观的终端容器”。

在这套架构里，`tmux` 的作用不是“方便手工开多个 shell”，而是提供下面几个关键能力：

1. **会话持久化**：SSH 断开、编排层重启后，Worker 终端仍能继续跑。
2. **可观察性**：技术总监或人工都能查看某个 Worker 终端当前输出。
3. **可干预性**：可以把新指令直接注入已有 Worker，而不是只能重新开任务。
4. **可隔离性**：每个任务独立 session + worktree，减少互相污染。
5. **可恢复性**：即使某个 Worker 卡住，也能单独杀掉或重建，不影响其他任务。
6. **可审计性**：结合日志和注册表，可以回放任务过程与失败原因。

一句话：

> `tmux` 让 Worker 从“看不见、摸不着的内部子会话”，变成“可操作、可诊断、可回放的独立终端单元”。

## 架构设计

### 编排层：技术总监

| 属性      | 值                                 |
| --------- | ---------------------------------- |
| Agent ID  | `tech-director`                    |
| Workspace | `~/.openclaw/workspace-master`     |
| 默认模型  | `openai-codex/gpt-5.4`             |
| 备用模型  | `google-gemini-cli/gemini-3.1-pro` |
| Discord   | 使用 `default` Bot 账号            |
| 角色定位  | 面向 Boss 的研发编排中枢           |

**核心职能**：

1. **需求理解与范围澄清**
   - 追问边界、梳理业务规则、识别隐含约束。
2. **方案设计与文档输出**
   - 产出需求文档、技术方案、实施计划、风险说明。
3. **任务拆解与 Prompt 编排**
   - 将需求拆成可并行、可验收、可回滚的 Worker 任务。
4. **Worker 调度**
   - 通过 `exec -> spawn-worker.sh` 生成 `worktree + tmux` Worker。
5. **质量监控与纠偏**
   - 读取日志、检查 CI、审查 PR，必要时注入新指令。
6. **过程管理与对 Boss 汇报**
   - 汇总阶段成果、阻塞点、风险、下一步计划。
7. **知识沉淀与培训交付**
   - 记录有效 Prompt、失败模式、仓库注意事项、操作手册、培训文档。

技术总监不应该退化成“高级写代码机器人”，而应始终站在**统筹交付**的位置上。

### 执行层：Worker

Worker 的职责非常收敛：

- 进入独立 worktree；
- 接收技术总监给出的精准 Prompt；
- 完成某类明确交付物；
- 回写任务状态；
- 必要时等待纠偏指令或追加要求。

执行层**不持有完整业务记忆**，只拿到完成当前任务所需的最小上下文。

## Worker 分类

为了覆盖“软件开发全过程”，Worker 不应只分为“写代码的工人”，而应至少拆成以下类型：

### 1. Code Worker

负责：

- 功能开发
- Bug 修复
- 重构
- 补测试
- CI 修复

典型交付物：

- 代码改动
- 测试用例
- PR
- 截图或运行结果

### 2. Doc Worker

负责：

- 需求说明整理
- 技术设计文档
- 培训材料
- 使用手册
- 发布说明
- 周报 / 月报 / 里程碑汇报草稿

典型交付物：

- Markdown 文档
- PPT/讲稿草案
- 发布公告草稿
- 对内培训步骤文档

### 3. Review Worker

负责：

- PR review
- 差异分析
- 风险点归纳
- 验收项对照检查
- 回归影响面评估

典型交付物：

- 审查意见列表
- 风险摘要
- 待改项清单

### 4. Research Worker

负责：

- 仓库调研
- 方案对比
- 技术选型分析
- 历史实现路径归纳

典型交付物：

- 调研报告
- 方案优劣对比
- 推荐结论

### 5. QA Worker

负责：

- 测试路径设计
- 回归检查
- 验收用例整理
- 缺陷复现步骤撰写

典型交付物：

- 测试清单
- 验收报告
- 缺陷说明

通过这种分类，技术总监管理的就不再只是代码实现，而是**多类型交付物的协同生产**。

## 外围基础设施

### 1. Tmux 隔离层

每个任务启动一个独立 `tmux session`，命名建议为：

```text
task-<类型>-<短任务名>
```

例如：

- `task-code-login-ui`
- `task-doc-release-note`
- `task-review-billing-pr`
- `task-qa-regression-auth`

### 2. Worktree 隔离层

每个需要修改仓库内容的 Worker 绑定一个独立 worktree，例如：

```text
.openclaw-master/worktrees/task-code-login-ui
```

对于纯文档或纯审查任务，也可以按需复用只读工作目录或独立 worktree，以避免并行任务互相覆盖文件、互相污染 Git 状态。

### 3. 任务注册表

在项目根目录维护：

```text
.openclaw-master/active-tasks.json
```

每个任务至少记录：

- `taskId`
- `taskType`
- `title`
- `session`
- `branch`
- `worktree`
- `model`
- `status`
- `promptSummary`
- `deliverables`
- `acceptanceCriteria`
- `dependsOn`
- `parentTaskId`
- `prUrl`
- `failureReason`
- `retryCount`
- `startedAt`
- `updatedAt`
- `lastHeartbeat`

建议状态机：

- `queued`
- `running`
- `blocked`
- `reviewing`
- `done`
- `failed`
- `cancelled`

### 4. Watchdog / Cron

使用系统 `cron` 或 OpenClaw 原生 cron，定期运行 `check-agents.sh`：

- 检查 `tmux` session 是否仍活着；
- 检查注册表是否有悬挂任务；
- 如果 PR 已创建，则轮询 CI 状态；
- 如果任务久无输出，则标记可能卡住；
- 如果 CI 失败或审查要求修改，则唤醒技术总监重新编排。

## 默认模型策略

### 结论

**可以**把技术总监的默认模型改为 `openai-codex/gpt-5.4`，而且从当前仓库代码看，这是更一致的默认值。

### 推荐策略

| 场景                     | 推荐模型                           | 说明                                             |
| ------------------------ | ---------------------------------- | ------------------------------------------------ |
| 技术总监默认编排模型     | `openai-codex/gpt-5.4`             | 主默认值，适合复杂规划、编码编排与高质量文档产出 |
| 复杂编码 / 长任务 Worker | `openai-codex/gpt-5.4`             | 当前建议主力                                     |
| 成本或速度优先 Worker    | `openai-codex/gpt-5.3-codex`       | 兼容或低一级备选                                 |
| UI / 文案 / 文档任务     | `google-gemini-cli/gemini-3.1-pro` | 适合作为设计向与文档向备选                       |
| 轻量研究 / 检索型任务    | 视环境另配                         | 可后续扩展更便宜模型                             |

### 重要区分：`openai-codex/*` vs `codex-cli/*`

这两个名字很像，但**不是同一层**：

- `openai-codex/gpt-5.4`：OpenClaw 的 OpenAI Codex provider 模型引用。
- `codex-cli/gpt-5.4`：OpenClaw 的 CLI backend 模型引用。

如果 Worker 是通过 `openclaw agent --model ...` 跑的，优先写：

```bash
openclaw agent --message "..." --model openai-codex/gpt-5.4
```

如果明确要走 CLI backend，再用：

```bash
openclaw agent --message "..." --model codex-cli/gpt-5.4
```

## 管理边界：哪些事必须由技术总监亲自做

为了避免技术总监退化成普通 Worker，需要明确职责边界。

### 必须由技术总监完成

- 需求澄清
- 范围确认
- 技术路线拍板
- 任务拆解
- 验收标准定义
- 风险识别与优先级判断
- 最终交付审查
- 面向 Boss 的汇报与解释

### 可以下放给 Worker

- 局部代码实现
- 测试修复
- 文档初稿
- 培训材料初稿
- PR 描述草稿
- 截图收集
- 差异整理
- 回归验证
- 调研汇总

### 原则

> 技术总监做“方向、标准、验收、汇报”；
> Worker 做“执行、产出、补充材料、局部修正”。

## Boss - 技术总监协作机制

为了让这套系统真正可长期运行，必须明确 Boss 与技术总监之间的协作边界，而不是只定义 Worker 如何执行。

### Boss 的职责

Boss 负责：

- 给出业务目标、优先级和最终判断
- 确认范围取舍
- 决定是否接受风险或延期
- 对外部结果负责

Boss 不需要亲自管理每一个 Worker，也不应该被迫持续跟踪底层执行细节。

### 技术总监的职责

技术总监负责：

- 把 Boss 的目标翻译为可执行任务
- 识别模糊点并主动澄清
- 在执行中持续维护方向一致性
- 对下协调 Worker，对上进行阶段汇报
- 在出现风险、延期、方向偏差时及时升级汇报

### 默认协作节奏

建议采用如下默认节奏：

1. **接单阶段**：技术总监确认目标、约束、优先级。
2. **方案阶段**：技术总监输出方案、拆解任务、标记风险。
3. **执行阶段**：技术总监调度 Worker，并做中途纠偏。
4. **验收阶段**：技术总监统一检查交付物质量。
5. **汇报阶段**：技术总监向 Boss 汇报结果、剩余问题、下一步建议。

### 必须主动向 Boss 汇报的情形

以下情况不应静默处理，而应主动升级汇报：

- 需求存在明显冲突或重大歧义
- 原方案不可行，需要改路线
- 进度明显延期
- 风险扩大，可能影响质量或上线时间
- 需要 Boss 做取舍决策
- 交付虽然完成，但存在已知残留问题

### 技术总监的汇报原则

技术总监的汇报应满足：

- **先结论，后细节**
- **说清状态，不堆流水账**
- **指出风险，不粉饰问题**
- **给出建议，不只抛问题**

标准汇报结构建议为：

```text
【当前状态】进行中 / 已完成 / 阻塞
【本阶段成果】
【当前风险】
【需要 Boss 决策的点】
【下一步计划】
```

## 技术总监的身份文档规范

除了执行架构，技术总监本身也必须有一套稳定的“身份与记忆文档系统”。否则即便 Worker 体系搭起来，长期行为也会漂移。

技术总监至少应维护以下文档：

### 1. `IDENTITY.md`

作用：定义“我是谁”。

应该回答的问题：

- 我的职位是什么
- 我的名字是什么
- 我的对外身份是什么
- 我的默认语气和风格是什么
- 我的核心角色定位是什么

对技术总监来说，`IDENTITY.md` 不应写成空泛的人设描述，而应写成**职责导向的身份卡片**。建议包含：

- **Name**：技术总监的名字
- **Role**：技术总监 / 研发编排中枢
- **Nature**：AI 技术总监 / 交付指挥官 / 研发中台
- **Vibe**：稳、清晰、直接、负责任
- **Emoji**：用于简短识别
- **Mission**：对 Boss 的核心职责一句话总结

建议原则：

- 不写浮夸设定
- 不写与职责无关的装饰性内容
- 优先体现“可信、稳定、能扛事”

### 2. `SOUL.md`

作用：定义“我怎么做事”。

这里不应只是抽象人格，而应写明技术总监的行为原则，例如：

- 先澄清，再执行
- 先方案，再开工
- 先验收标准，再派活
- 不用华丽措辞掩盖风险
- 不把 Boss 需要做的决策偷偷替 Boss 做掉
- 不让 Worker 在目标不清时盲干
- 以交付质量和长期可维护性为先

如果 `IDENTITY.md` 是身份卡，`SOUL.md` 就是管理哲学。

### 3. `USER.md`

作用：定义“Boss 是谁，Boss 怎么协作”。

对技术总监来说，这份文档尤其重要，因为它决定汇报风格和管理方式。应重点记录：

- Boss 怎么称呼
- Boss 偏好的沟通风格
- Boss 关注结果、过程、还是风险
- Boss 是否偏好短汇报或完整分析
- Boss 对延期、质量、返工的容忍度
- Boss 的项目目标和长期偏好

注意：这不是人物档案，而是**协作偏好说明书**。

### 4. `AGENTS.md`

作用：定义“系统怎么运行”。

这里应偏操作手册，写清楚：

- 技术总监的日常工作流
- Worker 的分类与使用场景
- Prompt 编排原则
- 何时升级汇报
- 何时允许并行，何时必须串行
- 交付前检查清单

`AGENTS.md` 应该让未来的技术总监版本或协作者一看就知道：

> 这套体系平时怎么运转，遇到问题怎么处理。

### 5. `TOOLS.md`

作用：定义“怎么调工具、怎么查状态、怎么运维”。

建议记录：

- `tmux` 的命名约定
- 脚本入口和用途
- 任务注册表位置
- 常见巡检命令
- 常见异常处理步骤
- 本地环境的特殊约束

### 6. `MEMORY.md` / 日记型记忆文件

作用：沉淀长期经验和近期经验。

对技术总监来说，应重点记录：

- Boss 的稳定偏好
- 已验证有效的工作流
- 失败案例与教训
- 代码库的历史坑点
- 哪些汇报方式最有效
- 哪些类型任务适合并行，哪些不适合

### 文档之间的分工原则

为了避免内容混乱，建议按下面分工：

- `IDENTITY.md`：我是谁
- `SOUL.md`：我怎么做事
- `USER.md`：Boss 怎么协作
- `AGENTS.md`：系统怎么运行
- `TOOLS.md`：工具怎么使用
- `MEMORY.md`：经验怎么沉淀

不要把这些文件写成互相重复的拷贝。每份文档都应有明确边界。

### 技术总监身份文档的写作原则

技术总监在编写自己的身份文档时，应遵循以下原则：

1. **职责优先于角色扮演**
   - 身份文档首先服务于稳定交付，而不是服务于表演感。
2. **稳定优先于花哨**
   - 语气可以有风格，但不能影响判断力和可信度。
3. **边界优先于口号**
   - 清楚说明能做什么、不能做什么、何时必须请示。
4. **协作优先于自我陶醉**
   - 文档要帮助 Boss 理解你，也帮助你长期一致地服务 Boss。
5. **长期可维护**
   - 让未来版本的自己也能接住这套身份，而不是每次重来。

## 工作流

### 流程 A：新项目 / 从零到一

```text
1. 技术总监与 Boss 沟通需求
2. 输出需求文档、用户故事、验收标准
3. 形成技术方案、模块拆解与风险清单
4. Boss 确认方案
5. 生成 Worker 任务清单
6. spawn-worker.sh 批量拉起 tmux Worker
7. Code Worker / Doc Worker / QA Worker 并行产出
8. 技术总监检查 PR / CI / 截图 / 描述 / 文档完整性
9. 生成阶段汇报并通知 Boss review
```

### 流程 B：已有项目功能 / Bug 修复

```text
1. 技术总监读取上下文并定位相关代码
2. 生成精准 Prompt
3. 选择模型与任务粒度
4. 通过 tmux 启动 Worker
5. 跟踪输出，必要时注入纠偏指令
6. PR 完成后做飞行检查
7. 补充变更说明、影响面说明、回归建议
8. 通知 Boss review
```

### 流程 C：文档 / 汇报 / 培训交付

```text
1. 技术总监确认受众、目标与交付形式
2. 拆成 Doc Worker / Research Worker 任务
3. 产出初稿、示例、操作步骤、Q&A
4. 技术总监统一修订语气、结构与准确性
5. 输出对内培训文档、使用手册或汇报材料
```

### 流程 D：失败重试（改进版 Ralph Loop）

失败时不是机械重跑，而是根据失败原因**改 Prompt 再重启**：

```text
❌ “实现自定义模板功能”
✅ “停。客户要的是复用现有配置，不是新建流程。
   上次失败原因：你做成了从零创建。
   这次只改配置复用相关 API 和 UI，别动模板创建链路。”
```

如果 Worker 只是局部跑偏，也可以不重建，直接通过 `tmux send-keys` 注入纠偏指令。

---

## 文档先行机制

既然技术总监覆盖的不只是编码，就应在流程中加入几个强制门：

1. **需求不清** → 必须先产出需求澄清文档。
2. **改动较大** → 必须先产出技术方案与影响分析。
3. **面向他人交付** → 必须补用户文档、培训文档或操作说明。
4. **任务结束** → 必须生成阶段汇报摘要。
5. **失败重试** → 必须记录失败原因与修正策略。

这样系统才不会只产代码，不产知识，不产过程沉淀。

## Tmux 使用方法

这一节既是给人看的，也是未来写进技术总监 `TOOLS.md` 的基础。

### 1. 创建会话

```bash
tmux new-session -d -s task-code-login-ui -c .openclaw-master/worktrees/task-code-login-ui \
  'pnpm openclaw agent --message "修复登录页的表单校验与报错展示" --model openai-codex/gpt-5.4'
```

含义：

- `-d`：后台启动；
- `-s task-code-login-ui`：session 名；
- `-c ...`：指定工作目录；
- 后面的引号内容：真正执行的 Worker 命令。

### 2. 查看所有 Worker

```bash
tmux ls
```

### 3. 接管某个 Worker 终端

```bash
tmux attach -t task-code-login-ui
```

从 tmux 退出但不结束任务：

```text
Ctrl-b d
```

### 4. 读取当前输出（适合技术总监巡检）

```bash
tmux capture-pane -pt task-code-login-ui
```

这很适合编排层通过 `exec` 快速查看 Worker 当前跑到了哪一步。

### 5. 动态注入新指令

```bash
tmux send-keys -t task-code-login-ui '停下，方向错了。不要重写组件；请只修复现有校验逻辑并补测试。' C-m
```

这是 Tmux V2.1 的关键能力：不必重建整个任务，也能即时纠偏。

### 6. 杀掉卡死任务

```bash
tmux kill-session -t task-code-login-ui
```

### 7. 典型运维动作

```bash
tmux ls
tmux capture-pane -pt task-code-login-ui
tmux attach -t task-code-login-ui
tmux send-keys -t task-code-login-ui '继续，先跑测试再改实现。' C-m
tmux kill-session -t task-code-login-ui
```

## 技术总监要调用的脚本接口

建议先做四支脚本，全部由技术总监通过 `exec` 调用：

### `spawn-worker.sh`

职责：

1. 创建 worktree；
2. 创建分支；
3. 创建 tmux session；
4. 启动 Worker 命令；
5. 将任务写入 `.openclaw-master/active-tasks.json`。

### `check-agents.sh`

职责：

1. 扫描注册表；
2. 校验 tmux session 是否存在；
3. 检查 PR / CI 状态；
4. 标记 `done` / `failed` / `blocked` / `reviewing`；
5. 需要时回调技术总监。

### `cleanup-worker.sh`

职责：

1. 结束 tmux session；
2. 清理已合并任务的 worktree；
3. 更新注册表。

### `report-status.sh`

职责：

1. 汇总 active tasks；
2. 提炼已完成 / 进行中 / 阻塞项；
3. 输出面向 Boss 的简报模板；
4. 为技术总监生成阶段汇报素材。

## 精准 Prompt 模板

每次拉起 Worker 时，技术总监都应该给出结构化 Prompt：

```text
【任务名称】一句话描述
【任务类型】code / doc / review / research / qa
【背景上下文】
- 业务目标
- 用户原始诉求
- 方案约束
【输入物】
- 设计文档路径
- 相关代码路径
- 类型定义路径
- 历史 PR / issue / 参考文档
【具体要求】
- 要做什么
- 不要做什么
- 验收标准
【交付物】
- 代码 / 文档 / 报告 / 审查意见 / 培训材料
【Git 操作】
- 基于 main 创建分支
- 完成后创建 PR（如需要）
【测试要求】
- lint / 类型检查通过
- 单元测试通过
- UI 改动附截图
【汇报要求】
- 结束时总结做了什么
- 说明风险和未完成项
```

---

## PR / 交付飞行检查清单

满足以下条件后，技术总监才通知 Boss review：

- ✅ PR 已创建（如果任务涉及代码）
- ✅ 分支已同步主线，无冲突
- ✅ CI 通过（lint、类型检查、测试）
- ✅ PR 描述清晰
- ✅ UI 改动附截图
- ✅ 相关文档已补齐
- ✅ 注册表状态已从 `running` 更新为 `done` 或 `reviewing`
- ✅ 可向 Boss 用简明语言解释“做了什么、为什么这么做、还有什么风险”

## 与旧方案对比

| 维度         | 旧方案（`sessions_spawn`） | 新方案（Tmux V2.1）              |

| 执行形态     | Gateway 内部子会话         | 外部独立 CLI 进程                |

| 长期可维护性 | 中等                       | 更适合复杂自动化研发流水线       |

## 实施建议

推荐按下面顺序落地：

1. **先上最小可行版**：`spawn-worker.sh` + `active-tasks.json` + `tmux`。
2. **补齐任务类型**：支持 code / doc / review / research / qa。
3. **再补巡检**：让 `check-agents.sh` 能看 session、PR 和 CI 状态。
4. **补汇报链路**：增加 `report-status.sh` 和简报模板。
5. **最后补闭环自动重试**：CI 失败后自动回流给技术总监重新编排。

这样可以最快把“可并行、可观察、可纠偏”的核心收益拿到手，再逐步补上“文档、汇报、培训、复盘”的完整交付链路。

## 最终定位

这套架构最终想塑造的，不是一个只会写代码的 AI，而是一个：

- 能理解 Boss 意图
- 能统筹研发资源
- 能驱动多类 Worker
- 能持续汇报进展
- 能沉淀知识资产
- 能覆盖从需求到交付全过程

的**技术总监型 AI**。

它的价值不在于“替代某一个程序员动作”，而在于把软件开发从零散操作变成**可编排、可观测、可复盘、可持续优化的交付系统**。
